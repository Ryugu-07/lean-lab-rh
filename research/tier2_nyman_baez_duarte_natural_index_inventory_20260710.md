# Tier 2 Nyman-Beurling Baez-Duarte Natural Index Inventory

Date: 2026-07-10

Target: `T2.nyman.baez.duarte.natural.index.inventory`

## Question

Which Lean index type and reciprocal map should represent the Baez-Duarte natural-parameter
finite approximants before attempting a formal bridge to the restricted real-parameter predicate?

## Literature Anchor

- Baez-Duarte's 2002 strengthening records the classical Nyman-Beurling subspace generated by real
  parameters `a >= 1`, then proves the stronger version where the generating subspace is restricted
  to natural parameters. See
  [arXiv:math/0202141](https://arxiv.org/abs/math/0202141).
- The published EuDML entry summarizes the same point as restriction to positive integer dilations
  and notes a constructive approximation sequence. See
  [EuDML 252348](https://eudml.org/doc/252348).
- The 2005 general strong criterion again phrases the discrete side through dilations indexed by
  natural numbers. See [arXiv:math/0505453](https://arxiv.org/abs/math/0505453).

Project translation:

- The project kernel is `fractionalPartKernel a x = Int.fract (a / x)`.
- The classical Baez-Duarte term `rho (1 / (n * x))` therefore corresponds to the project parameter
  `a = 1 / n`.
- The existing restricted parameter set is
  `nymanBeurlingRestrictedParameterSet = {a : Real | 0 < a and a <= 1}`.

## Candidate Lean Index Types

`PNat` / `ℕ+`:

- Mathlib defines `PNat` as `{n : Nat // 0 < n}` and provides notation `ℕ+`.
- This is semantically clean and can be used later if notation improves readability.
- The notation adds one more import/notation layer and may hide the subtype proof when debugging
  casts.

`{n : Nat // 0 < n}`:

- This is definitionally the same shape as `PNat`, but keeps the positivity proof visible as
  `n.property`.
- It matches the existing restricted-parameter workflow, which already uses subtype-indexed finite
  coefficients and `Function.Embedding.subtype`.
- It keeps the next bridge small: prove positivity and upper bound of `((n : Real) ^ -1)`, then use
  a map into `nymanBeurlingRestrictedParameterSet`.

`{n : Nat // 1 <= n}`:

- Equivalent mathematically, but less convenient locally because `Nat.cast_pos` consumes `0 < n`
  directly.
- It may be useful for finite truncation ranges, not for the global index type.

Plain `Nat` plus a support condition:

- This pushes the zero case into every finite-sum statement.
- It is a poor fit for `Finsupp.embDomain`, because the intended map is not meaningful at zero.

## Recommendation

Use the transparent subtype

```lean
{n : Nat // 0 < n}
```

as the first natural index type. The reciprocal parameter map should be the bundled map

```lean
fun n => (((n : Nat) : Real) ^ -1)
```

viewed as an element of `nymanBeurlingRestrictedParameterSet`.

The local proof obligations are deliberately small:

- positivity follows from `Nat.cast_pos` and `inv_pos`;
- the upper bound follows from `1 <= n`, casting to `Real`, and `inv_le_one_of_one_le₀`;
- later finite-sum transport should use an embedding from positive natural indices into the
  restricted real parameter subtype, analogous to the already compiled
  `exists_real_finsupp_of_restricted_finsupp`.

## Rejected Routes

- Do not start with the full classical criterion or an implication to `RiemannHypothesis`; that would
  skip the project-local finite-sum bridge.
- Do not keep indexing by arbitrary restricted real parameters when testing the Baez-Duarte
  strengthening; that hides the arithmetic restriction.
- Do not use plain `Nat` as the coefficient index; the zero case would become permanent clutter.

## M0 Correction

Batch `BATCH-20260710-M0-02` confirmed that reciprocal indexing is correct but not sufficient for
statement alignment. Baez-Duarte's theorem uses the full `L2(0,infinity)` error. After restricting
to `(0,1)`, the missing tail is exactly the square of the reciprocal coefficient moment. Any M1
criterion statement must restore that term; the local positive-natural predicate alone is not the
published criterion.
